## 4. WHY A TOP STUDIO WILL NOT BUY THIS

### 4.1 Animation Director

- **No demonstrated animation quality.** Zero rendered character animation clips exist in the repo. The "acting" system produces the same 8 micro-actions for every scene.
- **No key pose library.** The generative key pose system doesn't exist. `generate_pose_beats` returns a template, not actual poses.
- **No dailies/review workflow.** The review loop is a 20-line stub that returns a placeholder iteration result.

### 4.2 Lead Animator

- **No inbetweening.** `rigTools.ts` has 14 `UNSUPPORTED_BY_VERSION` tools — all the tools an animator would use are stubs.
- **No lipsync pipeline.** `lipsyncTools.ts` — 4 of 7 tools throw immediately. The phoneme timing generator is a rule-based heuristic that counts vowels.
- **No performance-driven animation.** The retargeting core has real IK but it's dead code. The live pipeline uses `atan2` angle extraction only.

### 4.3 Rigging Supervisor

- **No rigging automation.** 14 of 25 rigging tools are `UNSUPPORTED_BY_VERSION`. `create_pegs`, `create_deformers`, `create_eye_system`, `create_constraint`, `create_hand_swaps` — all throw immediately.
- **Rig validation is fake.** `validate_naming` always returns success. `validate_angle_coverage` hardcodes `missingAngles: ['back_3q_left', 'back_3q_right']` regardless of input.
- **No .tpl templates.** Zero template files exist. `export_template` throws `UNSUPPORTED_BY_VERSION`.

### 4.4 Pipeline TD

- **No CI.** Zero `.github/` files. No lint gate, no typecheck gate, no test gate. Anything can break silently.
- **No reproducibility.** `.env.example` hardcodes `/Users/romanmolodyko/Documents/toon-boom-harmony-mcp` in `HARMONY_ALLOWED_ROOTS` — won't work on anyone else's machine.
- **Nobody verified the bridge works.** The Harmony bridge script has 6 mock-only commands and 4 invented attribute names. The TVG vector creation path has never produced a real .tvg file.
- **No job queue.** Batch processing is `mock_only` — `run_batch_scene_assembly` returns 3 hardcoded results without reading the plans directory.
- **No render farm integration.** Render tools only support local Harmony CLI rendering. No Deadline, Qube, or Tractor integration.

### 4.5 CTO

- **No SLA.** No uptime guarantee, no error rate documentation, no performance benchmarks.
- **No auth.** The reconstruction FastAPI service (port 8765) has zero authentication — any local process can submit reconstruction jobs. The helper relay server (port 8765) also has no auth.
- **No observability.** No metrics endpoint, no structured logging, no tracing, no health dashboard beyond a single `health_check` tool.
- **No backup/disaster recovery.** Version rollback exists only at the JSON manifest level — no project-level or database-level backup.
- **No air-gapped mode.** Requires OpenAI API for LLM features (which don't work anyway). Requires network for webhook notifications.

### 4.6 Security Department

- **Command injection**: `src/adapters/screenshot/index.ts:39,42,44` — `exec` with template string interpolation. Any caller-controlled `outPath` is an attack vector.
- **Path traversal**: `scripts/python/harmony_bridge.py:508-516` — `release_lock` deletes lock files from user-supplied `projectPath` without root validation. `clean_unused_substitutions` (L546-557) deletes .tvg files from arbitrary paths.

- **Secret handling**: API keys are in `.env` (gitignored but exists). `.env.example` does not leak secrets.

### 8.2 Why VLM/SAM Integration Alone Is NOT a Moat

Integrating a Vision-Language Model or Segment Anything Model is **necessary infrastructure, not a competitive advantage**. Every animation startup has access to SAM, GPT-4V, Claude Vision, and Gemini. The integration takes 2-3 weeks. The output is:
1. **Generic** — available to all competitors simultaneously
2. **Non-differentiating** — the quality of VLM labeling depends on domain-specific post-processing, not the VLM itself
3. **Easily replaced** — when a better VLM releases, everyone upgrades at the same time

**The real moat is the flywheel**: VLM labels → human animator corrects → correction stored → fine-tuned model improves → fewer corrections needed → more engaged studios → more data → better model. **This flywheel does not exist in this codebase.** The `Studio Learning Loop` (Feature 10) is the key to starting it, but it requires:
1. Production deployment (0 studios)
2. Correction tracking (0 corrections)
3. Fine-tuned model (no model exists)
4. Feedback integration (no pipeline exists)
